## 1. Database: Postgres on Neon

Postgres (`@payloadcms/db-postgres`) is the right fit:
- Trainings ↔ locations ↔ templates are relational — Postgres handles this cleanly.
- First-class Drizzle migrations.
- **Neon** recommended for hosting: branchable DBs pair well with Vercel preview deployments.

## 2. Data model

### What changed vs. Strapi

- `locationName` + `address` + `imageName` were duplicated per training row → extracted into a
  `locations` collection.
- Email template was selected via string matching on `trainingType + locationName` in
  `app/api/apply-training/route.ts` → replaced by a `confirmationTemplate` relationship on each
  location. The right template is enforced by data, not code.
- `imageName` was a `/public/` path string → now a proper Payload Media upload.

### Collections

**`locations`**
- `name` (text, required, unique)
- `slug` (text, required, unique) — e.g. `bogenhausen`, `olympiapark`
- `address` (text)
- `image` (upload → `media`)
- `confirmationTemplate` (relationship → `email-templates`, required, filtered to `applicant_confirmation` type)

**`trainings`**
- `date` (date + time, required)
- `discipline` (select: `Schwimmen` | `Laufen`, required)
- `location` (relationship → `locations`, required)
- `isDisabled` (checkbox, default false)
- `capacity` (number, optional)

**`email-templates`**
- `name` (text, unique, required) — e.g. `trial_registration_notification`
- `type` (select: `applicant_confirmation` | `admin_notification`)
- `subject` (text, required)
- `html` (code field, HTML, required) — placeholders: `{{name}}`, `{{event}}`, `{{date}}`, `{{addresse}}`, `{{email}}`

**`media`** — Payload built-in uploads.

**`users`** — Payload default; admin login only.

**Recommended later**: a `registrations` collection to persist applications instead of
fire-and-forget email.

### Step 4 — Rewrite the two API routes to use Payload's Local API

The Local API runs in-process — no HTTP round-trip, no token required.

**`app/api/event-options/route.ts`** (full replacement):

```ts
import { getPayload } from 'payload'
import config from '@payload-config'

export const dynamic = 'force-dynamic'

export async function GET() {
  const payload = await getPayload({ config })
  const start = new Date(); start.setDate(start.getDate() + 2)
  const end = new Date();   end.setDate(end.getDate() + 21)

  const { docs } = await payload.find({
    collection: 'trainings',
    where: {
      isDisabled: { equals: false },
      date: { greater_than_equal: start.toISOString(), less_than_equal: end.toISOString() },
    },
    sort: 'date',
    depth: 2,     // populates location + location.image + location.confirmationTemplate
    limit: 100,
  })
  return Response.json(docs)
}
```

**`app/api/apply-training/route.ts`** (key change — template lookup):

## 4. Known gotchas

- **Neon pooler URL**: use the `-pooler` connection string for serverless (Vercel functions). The
  direct URL is for local dev and migrations only.
- **`sharp` placement**: must be in `dependencies`, not `devDependencies` — already correct.
- **Drafts**: Payload has `versions: { drafts: true }` per collection if you need draft/publish.
  Not needed for trainings but worth knowing for future collections.
- **Admin auth**: Payload admin is auth-gated; the Local API bypasses auth entirely (server-side only).
